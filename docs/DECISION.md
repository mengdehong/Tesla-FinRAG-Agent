# Architecture Decision Records (ADR)

本文件记录了 Tesla-FinRAG 项目开发过程中涉及的关键架构决策及其演进过程。

---

## 1. 数据处理策略 (Data Ingestion)

**原始要求**：将 PDF 财报中的文本段落和表格数据提取出来。

**实际决策**：**混合分层摄入**
- 保留 PDF/HTML 用于提取“叙述性文本（如管理层讨论）”。
- 引入 SEC 官方的 **XBRL (companyfacts)** 结构化数据处理核心财务数字。

**决策理由 (Why)**：
1. **准确度保障**：纯开源工具（如 pdfplumber）解析无边框、跨页的复杂财务报表准确率极难达到 100%。在金融场景中，错位一行即会导致利润率等核心数据错误。
2. **消除幻觉**：引入 XBRL 意味着直接获取特斯拉官方提交的结构化 JSON 数据，从源头上消除了大模型在“读取”PDF 表格并进行计算时的数值误差。

## 2. 向量数据库选型 (Vector Database)

**原始建议**：推荐使用 ChromaDB 或 FAISS-CPU。

**实际决策**：使用 **[LanceDB](https://lancedb.com/)**。

**决策理由 (Why)**：
1. **时间属性支持**：项目需处理如“2022年 Q3”等带有强时间属性的查询。
2. **过滤性能**：LanceDB 基于列式存储（Arrow），支持 SQL 级别的元数据（Metadata）硬过滤。
3. **检索能力**：原生集成“全文检索 (BM25) + 向量检索”，相比 ChromaDB 配置更简洁，相比 FAISS 功能更丰富。
4. **轻量化**：无须 C/S 架构（如Qdrant、Pgvector）依赖，适合本项目的数据量级。

| 特性 | FAISS | ChromaDB | LanceDB |
| :--- | :--- | :--- | :--- |
| 元数据过滤 | 不支持 | 支持 (配置繁琐) | **原生 SQL 过滤** |
| 混合检索 | 需手动实现 | 支持 | **原生集成** |
| 存储格式 | 内存为主 | 存储驱动 | **列式存储 (Arrow)** |

## 5. 输入处理策略 (Input Processing)

**正在从【正则驱动】向【混合架构】演进**：

*   **初始策略**：完全基于正则表达式驱动的 Query Planner。
*   **当前策略**：采用 **混合驱动架构（LLM 结构化抽取 + 定向规则兜底）**。

### 为什么演进？
1. **LLM 增强泛化**：LLM Planner 将自然语言（含双语/隐式时间）转换为强类型 `QueryPlan` (JSON)，解决复杂提问的解析遗漏。
2. **规则兜底保障**：当 LLM 输出损坏或超时时，平滑降级至正则规则，保证系统稳定性。
3. **XBRL 词典对齐**：内置官方词典，将口语化提问（如“研发花销”）自动对齐至标准字段（`ResearchAndDevelopmentExpense`），降低幻觉。

> [!WARNING]
> **当前局限性**：
> 1. **延迟**：多出来的 LLM 调用增加了意图解析阶段的耗时。
> 2. **模型依赖**：不同模型对 JSON 生成及置信度输出的能力不一，弱模型可能频繁触发 Fallback。
> 3. **映射精度**：XBRL 概念映射高度依赖 Embedding 模型特性，切换模型需重新标定阈值。
