## 1. 数据源
原始要求：将 PDF 财报中的文本段落和表格数据提取出来。
实际取舍：保留 PDF/HTML 用于提取“叙述性文本（如管理层讨论）”，但引入了 SEC 官方的 XBRL (companyfacts) 结构化数据来处理核心财务数字。
Why：
1. 纯靠开源工具（如 pdfplumber）解析无边框、跨页的复杂财务报表，准确率极难达到 100%。只要错位一行，算出来的利润率就是错的。
2. 解决幻觉： 真实的金融业务线不会用大模型去“读”PDF里的财务表格然后做数学题。引入 XBRL 意味着直接拿到了特斯拉官方提交的结构化 JSON 数据，消除数值抽取的误差。

## 2. 向量数据库的取舍
原始要求： 推荐使用 chromadb 或 faiss-cpu。
实际取舍： 使用LanceDB。
Why：
1. 题目明确要求处理“跨年趋势”和“2022年 Q3”等带有强时间属性的查询。
2. FAISS 是纯向量库，不支持元数据（Metadata）过滤。
3. ChromaDB 的多路混合检索（Hybrid Search）配置繁琐。
4. LanceDB 是底层的列式存储（Arrow），支持 SQL 级别元数据硬过滤，并且在一个库里原生集成了“全文检索(BM25) + 向量检索”。
5. pgvector/Qdrant 等数据库大多为C/S架构，依赖沉重，且该项目的数据量级并不需要。

## 3. 多步推理逻辑的取舍
原始要求： 设计一个能够处理多步推理的问答流程。
实际设计：并未设计超长Prompt, 而是引入`Tools` 的概念，引入代码工具 查询规划器和数值计算器等
Why：
1. 大模型做复杂算术题（如“计算四个季度研发费用总和”）时易算错。
2. 引入通用MCP工具，未必能处理好该项目的专业财务问题，且MCP工具为C/S架构需要额外维护成本。


## 4. AI开发框架的取舍
实际设计并未采用LangChian等成熟的Agent框架，而是采用自己设计的、基于函数调用的工具使用框架。
Why：
1. LangChian等框架虽然功能丰富，但隐藏了大量细节，难以后续针对财务领域的特殊需求进行定制化改造。
2. 手写核心层更透明、易维护
